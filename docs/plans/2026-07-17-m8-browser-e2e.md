# M8 — Browser End-to-End Implementation Plan

> **For agentic workers:** execute task-by-task in order; steps use checkbox (`- [ ]`) syntax for tracking. Every task ends green and committed before the next starts.

**Goal:** Six Playwright journeys driving real Chromium against the real stack (web, API, PayMock, Gate, Postgres, Redis), green on every pull request — per `docs/specs/2026-07-17-m8-browser-e2e-design.md`.

**Architecture:** A new `tests/e2e` workspace package owns a Playwright config whose `webServer` array boots all four services on localhost. Postgres and Redis come from compose locally and from service containers in CI. `globalSetup` reseeds so the suite is repeatable; `workers: 1` because every spec shares one seat inventory. Tests locate controls by accessible role and name — never `data-testid` — so the suite doubles as an accessibility assertion.

**Tech Stack:** `@playwright/test` (Chromium only), the existing `db:seed`, `POST /api/demo/login` for auth state, PayMock's real hosted pay page, GitHub Actions service containers.

## Global Constraints

- **No code comments** in any file or snippet (repo rule; reasoning goes in ADR 0010).
- Conventional Commits, English code/docs/commits, **no AI attribution** in commit messages.
- Never hardcode an id: `db:seed` regenerates every event, seat, and ticket id on each run. Slugs (`bangkok-indie-fest`, `midnight-drop`) are stable; resolve everything else through the API.
- Money is integer satang; a hold is a "hold", never a "lock" or "reservation" (`CONTEXT.md`).
- The package's test script is named **`e2e`**, never `test` — `turbo run test` runs in the `quality` CI job, which has no stack behind it.
- Existing gates stay green: `pnpm turbo run lint typecheck build test` and the 41 jest e2e specs are untouched by this milestone.
- A global `GqlThrottlerGuard` rate-limits **REST too**: 120 req/60s per IP (`THROTTLE_LIMIT`), and `POST /api/demo/login` is capped at 20/60s. Six serial journeys sit far under both, but a spec that loops requests will meet a 429 that looks like a product bug and is not. If one ever does, that is the thing to check first.
- Milestone ends deployable: `APP_VERSION` → `m8`.

## Facts established by reading the code — do not re-derive

- Seat accessible name: `` `${section} ${rowLabel}${number} — ${mine ? "yours" : status}` `` → `Front A1 — available`. Statuses: `available | held | sold`; a seat you hold reads `— yours`. Element is `<rect role="button">` with a nested `<title>` of the same string.
- Seat checkout button: `Pick seats to continue` (none selected) → `Claim 1 seat` / `Claim 2 seats` → `Claiming…`.
- GA stepper: `aria-label="Add one General admission"` / `Remove one General admission`. Claim button: `Pick your tickets` → `Claim 1 ticket` → `Claiming…`.
- Seat 409 refusal surfaces as a **sonner toast**, then `refresh()`. Sonner renders each toast with `role="status"`.
- Order page: `awaiting_payment` → `<h1>Complete your payment</h1>`; `paid` → a `<Badge>paid</Badge>`; `expired`/`canceled` → an empty state titled `This order expired` / `This order was canceled`, plus toast `The payment was declined — your order was canceled`.
- PayMock pay page: `<form method="post" action="/pay/{id}/confirm">` with `<button name="outcome" value="success">Pay {{.Amount}}</button>` and `<button name="outcome" value="fail">Simulate a failed payment</button>`.
- Check-in: `<Label for="checkin-token">Ticket QR token</Label>` + `<Input id="checkin-token">` + `<Button type="submit">Check in</Button>`.
- Seed: `bangkok-indie-fest` (seated map `theater` — section **Front** rows A–D × 10, `150000` satang; section **Main** rows A–F × 12, `90000`; plus free GA `General admission` with `remaining: 176`, and a sold-out `Early bird`), `midnight-drop` (`dropMode`, free `Drop pass`, `maxPerOrder: 2`). Sold seats are deterministic: `index % soldEvery === 2`.
- Demo users `demo-buyer@openseat.dev` / `demo-organizer@openseat.dev` have `passwordHash: null` — UI login is impossible; `POST /api/demo/login` `{ role }` is the only route in.
- Ports: web `3000`, api `4000`, paymock `4100`, gate `4200`. Gate defaults: `ADMIT_BATCH=3`, `ADMIT_INTERVAL_MS=2000`, `ADMISSION_TTL_SECONDS=300`.
- A failed-payment webhook releases inventory **in the same transaction** (`payments.service.ts` `handleFailed` → `releaseOrderInventoryInTx(tx, orderId, 'canceled')`), so no test waits out the 15-minute window.

```ts
export async function issueFreeGaTicket(
  request: APIRequestContext,
  slug: string,
): Promise<{ orderId: string; qrToken: string; guestToken: string }> {
  const event = await getEvent(request, slug);
  const ga = event.ticketTypes.find((t) => t.name === "General admission");
  if (!ga) {
    throw new Error(`no General admission tier on ${slug}`);
  }
  const created = await request.post(`${API}/api/events/${event.id}/orders`, {
    headers: { "Idempotency-Key": randomUUID() },
    data: {
      items: [{ ticketTypeId: ga.id, quantity: 1 }],
      buyerEmail: `e2e-${randomUUID()}@openseat.test`,
      buyerName: "E2E Buyer",
    },
  });
  if (!created.ok()) {
    throw new Error(
      `order failed: ${created.status()} ${await created.text()}`,
    );
  }
  const order = (await created.json()) as {
    id: string;
    guestToken: string;
    tickets: Array<{ qrToken: string }>;
  };
  return {
    orderId: order.id,
    guestToken: order.guestToken,
    qrToken: order.tickets[0].qrToken,
  };
}
```

Add `import { randomUUID } from "node:crypto";` at the top of the file.

All verified against source, so this needs no adjustment at run time:

- `@Post('events/:eventId/orders')` (`orders.controller.ts:24`, declared on a prefix-less `@Controller()`), guards `OptionalJwtAuthGuard, AdmissionGuard`. The admission guard only bites when `event.dropMode === true`, so `bangkok-indie-fest` needs no token.
- `CreateOrderDto`: `items?: [{ ticketTypeId, quantity }]` **or** `seatIds?: string[]` — both optional individually, one required. **`buyerEmail` and `buyerName` are required**, which is the field most likely to be dropped by mistake. `ValidationPipe` runs with `whitelist: true`, so any extra field is silently stripped rather than rejected.
- Headers read: `idempotency-key`, `x-hold-key` (seat orders only), `x-admission-token` (drop events only).
- A free order (`totalSatang === 0`) is created `status: 'paid'` with its tickets in the **same transaction**, then returned through `ORDER_INCLUDE` — so `tickets[].qrToken` and the top-level `guestToken` are both on this response. No follow-up GET. (For the record, the guest read route is `GET /api/orders/:id?token=<guestToken>`.)

```ts
import { test as base, type Browser, type BrowserContext } from "@playwright/test";

export const WEB = "http://localhost:3000";

export const test = base.extend({
  context: async ({ context }, use) => {
    await context.addCookies([
      { name: "os_locale", value: "en", url: WEB },
    ]);
    await use(context);
  },
});

export const expect = base.expect;

export async function demoContext(
  browser: Browser,
  role: "buyer" | "organizer",
): Promise<BrowserContext> {
  const context = await browser.newContext();
  await context.addCookies([{ name: "os_locale", value: "en", url: WEB }]);
  const response = await context.request.post(`${WEB}/api/demo/login`, {
    data: { role },
  });
  if (!response.ok()) {
    throw new Error(
      `demo login (${role}) failed: ${response.status()} ${await response.text()}`,
    );
  }
  return context;
}
```

**This posts to the web origin, not the API origin — the distinction is load-bearing.** `AuthProvider` calls `refreshSession()` on mount and derives the access token from the httpOnly refresh cookie, so a context holding that cookie is a logged-in context; the token in the login response body is never needed. But the cookie is scoped to `path=/api/auth`, and the app requests `/api/auth/refresh` **through the Next rewrite on `:3000`**. Posting to `:4000` directly would happen to work — cookies key on domain and ignore port, so a `localhost:4000` cookie is sent to `localhost:3000` — but relying on that quirk would break the moment either service moved to a real hostname. Going through the proxy sets the cookie on the origin that will actually send it.

`POST /api/demo/login` is throttled at 20/60s. Six specs is nowhere near it.

- [ ] **Step 4: Write the seat helpers**

`tests/e2e/fixtures/seats.ts`:

```ts
import { expect, type Locator, type Page } from "@playwright/test";

export type SeatStatus = "available" | "held" | "sold" | "yours";

function escapeRegExp(value: string): string {
  return value.replace(/[.*+?^${}()|[\]\\]/g, "\\$&");
}

export function seat(page: Page, label: string): Locator {
  return page.getByRole("button", {
    name: new RegExp(`^${escapeRegExp(label)} —`),
  });
}

export async function expectSeatStatus(
  page: Page,
  label: string,
  status: SeatStatus,
): Promise<void> {
  await expect(seat(page, label)).toHaveAccessibleName(
    new RegExp(`^${escapeRegExp(label)} — ${status}$`),
  );
}

export async function firstAvailableSeat(
  page: Page,
  section: string,
): Promise<string> {
  const available = page.getByRole("button", {
    name: new RegExp(`^${escapeRegExp(section)} \\w+\\d+ — available$`),
  });
  await expect(available.first()).toBeVisible();
  const name = await available.first().getAttribute("aria-label");
  if (name === null) {
    throw new Error(`no available seat in section ${section}`);
  }
  return name.split(" — ")[0];
}
```

The prefix regex is deliberate: a seat's accessible name carries its status, so `Front A1 — available` becomes `Front A1 — yours` the moment it is held. Anchoring on `^Front A1 —` keeps the locator valid across the transition; `expectSeatStatus` asserts the status separately. `^Front A1 —` also cannot match `Front A10 — …`, because the character after `A1` there is `0`, not a space.

- [ ] **Step 2: Write ADR 0010**

`docs/adr/0010-browser-tests-locate-by-role.md` — follow the house shape used by ADR 0009 (`Status` / `Context` / `Decision` / `Consequences` / `When this would change`). It must record:

- **Locate by role and name, not `data-testid`.** M6's accessibility pass left every control with a role and an accessible name, so the tests need nothing added to production markup — and the suite becomes an accessibility regression net: if Playwright can find a control the way a screen reader would, so can a screen reader. Cost: coupling to copy, contained by pinning `os_locale=en` on every context. Seat names carry status, so locators anchor on the label prefix and assert status separately.
- **Serial execution (`workers: 1`).** Every spec shares one seat inventory; parallel specs would contend for the very invariant under test. Matches the API suite's `maxWorkers: 1`. Cost: wall clock. The move, if the suite grows: partition seats per spec.
- **`globalSetup` reseeds.** Repeatability over isolation, since `db:seed` already hard-deletes and recreates the demo events. Cost: running the suite locally discards demo-event state.
- **PayMock is not stubbed.** Its pay page and its deliberately double-sent webhooks are the most interesting seam in the system; stubbing them would delete the reason the journey is worth driving in a browser.
- **`reuseExistingServer: false`, against Playwright's own convention.** Discovered in Task 1, not designed: `reuseExistingServer: !process.env.CI` is the documented default, and on its first run it reused an unrelated Next.js app already listening on port 3000 and asserted against that app's login page. Playwright identifies a server by whether *anything* answers on the port — that is liveness, not identity. The suite now always owns its stack, and a busy port is a loud startup failure instead of a silent wrong-application test. Cost: `pnpm dev` cannot be running during a local `pnpm e2e`. Worth recording because the failure mode is invisible when the assertion is loose, which is exactly when a suite is least trustworthy.
- **When this would change:** a second writer to the same inventory (a parallel CI shard, a shared staging database) breaks the `workers: 1` assumption; per-spec seat partitioning is the answer, not per-spec databases.

## Self-Review

**Spec coverage.** Every spec section maps to a task: decisions 1–3 (locators, locale, prefix) → Task 2 + ADR in Task 9; decision 4 (own package, `e2e` script) → Task 1; decision 5 (`workers: 1`) → Task 1 config + ADR; decision 6 (reseed) → Task 1 global setup; decision 7 (prod build) → Task 1 `webServer` uses `start`/`start:prod`; decision 8 (no inter-test deps) → Task 7 issues its own ticket; decision 9 (PayMock live) → Tasks 3 and 5. Six journeys → Tasks 3–7. CI → Task 8. All six deliverables → Tasks 1, 8, 9.

**Placeholders.** None. Every code step carries the code; every verification step carries the command and the expected result. Where a fact could not be established without a running stack — the event route's response shape, the guest-field labels, the queue page's copy, the 409 toast text — the step says exactly what to run to establish it and to correct the code to match, rather than leaving a TODO.

**Type consistency.** `seat`/`expectSeatStatus`/`firstAvailableSeat` are defined in Task 2 and used with those names in Tasks 3, 4, 5. `getEvent`/`issueFreeGaTicket`/`demoContext` are defined in Task 2 and used in Task 7. `EventSummary` is the one shared type. `SeatStatus` covers exactly the four names the markup can produce (`available | held | sold | yours`).

**Assumptions verified before execution, not during.** The plan originally left four things to be checked at run time. All four were settled against the source first, and one was wrong:

- `GET /events/:slug/ticket-types` — **did not exist.** Ticket types are embedded in the event response. `issueFreeGaTicket` was rewritten; as first drafted it would have failed on its first run.
- `POST events/:eventId/orders` and its DTO — correct, with `buyerEmail`/`buyerName` required and `whitelist: true` silently stripping extras.
- Free GA orders return `tickets[].qrToken` and `guestToken` on the create response — correct, so Task 7 needs no follow-up fetch.
- `demoContext` — resolved: `AuthProvider` rehydrates from the refresh cookie on mount, so the cookie alone is a session. The fixture posts through the web origin rather than the API origin, so it does not depend on cookies ignoring ports.

**Remaining soft spots, honestly.** Three specs name copy that could not be confirmed without a running stack: the 409 refusal toast (Task 4, Step 3), the guest name/email field labels (Task 5, Step 2), and the queue page's controls (Task 6, Step 1). Each of those steps says exactly what to read and to correct the spec to match. They are cheap to resolve at execution and expensive to guess at now.
