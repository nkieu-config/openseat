# ADR 0011: Refunds reclaim the seat first and settle the money second

Status: Accepted (2026-07-18)

## Context

M3 proved this system can take money honestly: an order commits, PayMock hosts the payment, and a signed, deliberately duplicated webhook settles it through dedup and a single transaction. M9 gives the money back, and a refund is not a purchase run backwards through the same code — it inverts which side is authoritative and which side is compensating.

A purchase is *money-first*: nothing the buyer owns exists until the provider confirms payment, so the webhook is where tickets are born. A refund is the mirror: the organizer already owns the decision, and the two effects it triggers — voiding the ticket and returning the money — have opposite reversibility. Restoring a seat to inventory is a local, certain act. Moving money is a remote, fallible one. The order in which they happen decides what a mid-flight failure leaves behind, and only one order leaves behind something recoverable.

Three questions had to be answered before the first line: which effect commits first, where each ledger settles, and how the settlement webhook finds the refund it belongs to. The third turned out to hide a race that would have stranded real money.

## Decision

**Reclaim first, settle second.** Transaction 1, owned by the API, runs the instant the organizer asks: it voids the chosen tickets, restores GA `remaining` and frees seats, and writes the `seats.released` outbox row that pushes the map live. The provider refund call follows *after* that transaction commits; the `payment.refunded` webhook settles the money accounting later, through the same HMAC + dedup machinery every other webhook uses.

The alternative — call the provider first, void tickets only once the money is confirmed — was rejected on its failure mode. Compare the two the way the runbook has to:

- **Settle-first, provider call fails:** the money never moved, but if the void had gone first it would now need clawing back — a seat may already be re-held or re-sold to someone else. There is no safe undo. The system is wedged between two truths.
- **Reclaim-first, provider call fails:** the ticket is void, the seat is back on sale, and the money leg is marked `failed` with a Retry button. Nothing is inconsistent; one recoverable step remains. Re-sending money is idempotent on the provider side; re-selling a seat is not idempotent on ours.

The asymmetry is the whole argument: **the irreversible effect must be the one we own and can guarantee, and the fallible effect must be the one that can be retried without harm.** Reclaim-first is the only ordering where a partial failure is a pending task rather than a corruption.

**Each ledger settles where it is owned.** The refund's *inventory* truth (tickets void, seats free) settles in Transaction 1 the moment the organizer acts, because that is a fact about our own database and waiting on a remote provider to make a seat purchasable again would be a lie about who controls inventory. The refund's *money* truth (`refundedSatang`, order status) settles only when `payment.refunded` arrives, because that is a fact the provider owns and asserting it before the provider confirms would be the same lie M3 refused to tell on the way in. A refund therefore lives briefly in a legitimate split state — ticket already void, money still `pending` — and that is correct, not a window to close.

**PayMock gets exactly what a real provider has, and nothing about tickets.** A refund endpoint, cumulative refund state on the intent (`RefundedSatang`, guarded against over-refund), and one new event type shipped through the existing dispatcher. Three richer designs were rejected for breaking the simulation's honesty:

- **Refund logic inside PayMock** — letting it know about tickets or seats. Rejected: the provider is a payment vendor; the day it understands inventory it stops standing in for Stripe and starts hiding the coordination problem that is the point of the exercise.
- **A dedicated refunds microservice.** Rejected: there is no second writer, no independent scaling pressure, and no bounded context that `refunds` inside the monolith does not already express (ADR 0001). It would be architecture cosplay.
- **A durable queue (Kafka/SQS) for the money leg**, to get retries for free. Rejected at this scale for the same reasons as ADR 0003: the outbox plus the organizer's Retry button already make the leg recoverable, and the webhook's own backoff covers the transient case. The queue buys durability we can already assert without it.

**The webhook matches on the caller's `reference`, not the provider's refund id — because of a real race.** PayMock's dispatcher fires from a goroutine the instant the refund is accepted, so the `payment.refunded` webhook can reach the API *before* the synchronous refund response has returned and been stored. If settlement matched on PayMock's own `refundId`, the early webhook would find no refund carrying that id yet, conclude there was nothing to settle, and strand the refund `pending` with the money already gone. So the API sends its *own* refund id as an opaque `reference` on the refund call, and PayMock echoes it back in the event — the Stripe-metadata pattern. Settlement matches on an id the caller minted and already knows, which cannot lose the race. This is why `reference` exists and why it is not merely `refundId` by another name: one is known before the provider is called, the other only after.

**Order status is derived from ticket counts, not from amount comparison.** At settlement, `refunded` vs `partially_refunded` is decided by counting tickets whose status is not `void`: zero live tickets means `refunded`, any remainder means `partially_refunded`. The design spec originally compared `refundedSatang` to the order total. That comparison lies for free orders — a fully-refunded free order has `refundedSatang == 0 == totalSatang`, indistinguishable from a half-refunded one, and both from an untouched one. Counting tickets is the only derivation that survives a zero-price order, and free orders are the common case here. The deviation from the spec is deliberate and is the reason this ADR, not the spec, is the authority on status derivation.

**No `refund.failed` webhook.** A real provider emits one; PayMock does not, and the API does not depend on one. A failed money leg is caught synchronously — the refund call throws, the leg is marked `failed`, and the organizer retries. Adding an asynchronous failure event would duplicate a signal we already have in-band and invite a second dedup path for no new capability. This is an accepted simplification, recorded so its absence reads as a decision rather than an oversight.

## Consequences

- **The order state graph grew two terminal-ish states**, `partially_refunded` and `refunded`, both reachable only from `paid`. `partially_refunded` is not terminal — it refunds again down to `refunded`. Every "was this order paid" read in the dashboard had to widen to the set `{paid, partially_refunded, refunded}`, and revenue became `sum(totalSatang) - sum(refundedSatang)` over that set. This widening reached further than the plan anticipated: the roster query, the gross-revenue aggregate at both call sites, and the paid-order count all shared the old `status: 'paid'` filter, and each would have silently dropped a partially-refunded order from the organizer's totals. The honest-revenue change is therefore larger than "subtract a column" — it is a consistent redefinition of *paid* across every organizer read.

- **The suite found two real bugs that only a full refund-then-resell cycle exercises**, both invisible to every test written before M9:
  - *A voided ticket kept its seat and blocked the resale.* The partial-unique backstop `UNIQUE (event_id, seat_id) WHERE seat_id IS NOT NULL` (ADR 0002) counted void tickets, because a refund keeps `seat_id` on the ticket so the buyer can still see which seat they lost. Re-*holding* the seat worked — the hold SQL already excludes `status <> 'void'` — but *issuing* the new ticket hit the constraint and 500'd inside the settlement webhook. The index now also excludes `status <> 'void'`, matching the hold SQL and the seat map, which have skipped void since M2. The lesson is that the three places that decide "is this seat taken" — hold SQL, seat-map read, and the unique index — must agree on what void means, and the index was the one that never got the memo.
  - *Reseeding died on any database that had a refund in it.* `db:seed` deletes orders, but a refund references its order under `onDelete: RESTRICT`, so the delete aborted and every reseed — local, CI e2e globalSetup, and production deploy — failed the moment refunds existed. Fixed by deleting refunds before orders in both cleanup blocks. This was a production bug, not a test bug; the browser journey surfaced it only because it was the first thing to leave a refund behind for the next reseed to trip over.

- **Refunds are the first genuinely dangerous organizer action**, and they ship under the same ownership check as every read — `organizerId === user.id`, returning 404 not 403. That is deliberately the weakest defensible authorization for an action that moves money, and it is exactly the pressure that justifies M10's RBAC: the mechanism now exists, so the question of *who* may fire it stops being hypothetical.

## When this would change

If a second payment provider is added, the `reference` echo assumption has to be re-checked against that provider's webhook-vs-response ordering — the race was PayMock's goroutine, but the *fix* is a contract we now depend on and a new provider might not honor it. If buyer self-service refunds arrive (M9 explicitly built mechanism, not policy), the ownership check is no longer sufficient and refund *policy* — window, fee, who may — becomes real product surface. And if the money leg ever needs to survive an API restart mid-flight rather than a mere provider hiccup, that is the point where the durable queue rejected above earns its place; the outbox already marks the boundary where it would slot in.
